# Search Typeahead System

A working **search typeahead / autocomplete** service, like the suggestion feature in
search engines and e-commerce sites. As the user types a prefix, the system returns
the most popular matching queries; it records submitted searches, updates query
popularity, supports **trending searches**, and uses a **distributed cache** for
low-latency reads.

## What it demonstrates (and where)

| Concept | File | One-line explanation |
|---|---|---|
| **Data modeling** | `app/db.py` (Postgres) | One table: `search_queries(query, count, last_searched_at)`. |
| **Suggestion engine** | `app/trie.py` | A Trie where each node caches its subtree's top candidates → O(prefix) lookups. |
| **Distributed cache** | `app/cache_cluster.py` | 3 Redis cache nodes; **consistent hashing routes each prefix to one node**. |
| **Consistent hashing** | `app/consistent_hash.py` | Hash ring with virtual nodes; adding a node remaps only ~1/N of prefixes. |
| **Trending / recency** | `app/activity.py`, `app/suggester.py` | Sliding minute-window counters; recency-aware ranking blends popularity + recent activity. |
| **Batch writes** | `app/batch_writer.py` | Search events are buffered and flushed as one bulk UPSERT. |

See **`DESIGN.md`** for the reasoning and trade-offs behind every choice, and
**`PERFORMANCE.md`** for measured latency, cache hit rate, and write reduction.

## Architecture

```
Browser (debounced typeahead UI)
   │  GET /suggest?q=…      POST /search {query}      GET /trending      GET /cache/debug
   ▼
FastAPI backend
   ├─ CacheCluster ── consistent-hash ring ──▶ cache-0 / cache-1 / cache-2  (Redis)
   │       (route prefix → owning node; hit returns instantly)
   ├─ Suggester ──▶ Trie (in-memory index, top-K per node)        ┐ basic = count
   │            └─▶ Activity (recent window, Redis-meta)          ┘ recency = count + α·recent
   └─ BatchWriter ── buffer ──flush──▶ Postgres (bulk UPSERT)
   ▼
Postgres  (source of truth: query → count)
```

## How to run

You need Docker. Then:

```bash
docker compose up --build
```

This starts Postgres, three Redis cache nodes, a Redis-meta node, and the backend. On
first startup it **generates a 100,000-query dataset** (`backend/data/queries.csv`),
ingests it, and builds the Trie. When ready, open:

- **App UI:**   http://localhost:8000/
- **API docs:** http://localhost:8000/docs

Stop and wipe everything (including the DB volume):

```bash
docker compose down -v
```

## Dataset

- **Source:** synthetic, generated by `backend/scripts/generate_dataset.py` (fixed
  seed → reproducible). It produces ≥100,000 unique queries with a heavy-tailed
  (Pareto) popularity distribution, in `query,count` CSV format.
- **Using your own dataset:** drop any `query,count` CSV at `backend/data/queries.csv`
  and restart with an empty DB (`docker compose down -v && docker compose up`).
  `app/db.py::ingest_csv` lowercases, aggregates duplicates, and bulk-loads it.

## API

| API | Purpose | Behaviour |
|---|---|---|
| `GET /suggest?q=<prefix>&ranking=basic\|recency&limit=10` | Fetch suggestions | Up to 10 prefix-matches sorted by count (basic) or recency-aware score. |
| `POST /search` `{"query": "..."}` | Submit a search | Returns `{"message": "Searched"}`, records the query (batched write + trending). |
| `GET /cache/debug?prefix=<prefix>` | Debug cache routing | Shows which cache node owns the prefix and whether it's currently cached (hit/miss). |
| `GET /trending?k=10` | Trending searches | Top queries in the recent sliding window. |
| `GET /metrics` | Performance counters | Cache hit rate, DB read/write counts, batch write-reduction factor. |
| `GET /stats` | Config + dataset size | Useful for the demo. |

### Quick API examples

```bash
curl "http://localhost:8000/suggest?q=iph&ranking=basic"
curl "http://localhost:8000/suggest?q=iph&ranking=recency"
curl -X POST http://localhost:8000/search -H "Content-Type: application/json" -d '{"query":"iphone 15"}'
curl "http://localhost:8000/cache/debug?prefix=iph"
curl "http://localhost:8000/trending?k=10"
curl "http://localhost:8000/metrics"
```

```bash
python -m scripts.benchmark 3000 # latency (p50/p95/p99) + cache hit rate
python -m scripts.generate_dataset data/queries.csv 100000   # regenerate the dataset
```

## Measured performance (see PERFORMANCE.md)

- Suggestion latency: **p95 ≈ 1.85 ms**, p99 ≈ 3.5 ms.
- Cache hit rate: **98.8%**.
- Batch writes: **120× fewer** DB write statements vs writing per event.
- Adding a cache node remaps only **~23%** of prefixes (≈ 1/N).
